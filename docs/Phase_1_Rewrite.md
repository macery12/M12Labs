# M12Labs Panel UI Rewrite — Phase 1

> Goal of the rewrite: **no babel, no reused code, secure, fast (runtime), fast (build).**
> Goal of Phase 1: a **fully navigable skeleton** — real landing page, real login page, the full app shell (top nav + all three sidebars), every V1 route wired up, and one shared `<Placeholder>` page rendered for every route not yet built.

---

## 1. Target stack (and why)

| Concern | Choice | Why |
|---|---|---|
| Renderer | **React 19** | Zero compat-layer surprises; full ecosystem; swap to `preact/compat` later via one Vite alias if bundle size ever matters |
| Build | **Vite 6 + `@vitejs/plugin-react-swc`** | SWC = Rust, **no babel**, fast HMR + builds |
| Language | **TypeScript (strict)** | Already typed; keep `models.d.ts` shapes |
| Styling | **Tailwind v4** (`@tailwindcss/vite`) + **Radix UI** primitives + **CVA** | Tailwind v4 has no babel + Rust engine; Radix gives accessible dialog/dropdown/tooltip/switch/select (maps onto V1 `elements/`); CVA replaces the styled-components variant pattern |
| Server state | **TanStack Query v5** | Caching/revalidation/loading/error for free; deletes most easy-peasy boilerplate |
| Client state | **Zustand** | Tiny global store for `user`, `theme`, `flashes`, `socket`, active-server |
| Routing | **react-router v7** (data router) | Keeps the V1 declarative-route pattern |
| Forms | **react-hook-form + zod** | Replaces Formik; less re-render, schema validation, no babel |
| HTTP | **axios** wrapper | Keep CSRF + Fractal transformers; one interceptor module |
| i18n | **i18next + react-i18next** | Keep existing string keys |
| Icons | **lucide-react** (or keep heroicons) | Tree-shakeable |

> **The reversible-framework trick:** write standard React. To trial Preact later, add `resolve.alias` for `react`/`react-dom` → `preact/compat` in `vite.config.ts`. Nothing else changes.

## 3. The route registry (drives nav AND placeholders)

One typed registry per area, mirroring V1's `route()`. Each entry carries `path`, a lazy `element` **or** `placeholder: true`, plus `name/icon/category/permission/condition`. Until a page is built, set `placeholder: true` and the router renders `<Placeholder title={name} />`.

```ts
// routes/registry.ts (shape)
export interface RouteDef {
  path: string;
  name?: string;
  icon?: IconType;
  category?: string;
  permission?: string;
  condition?: (f: Flags) => boolean;
  element?: React.LazyExoticComponent<React.FC>; // built page
  placeholder?: boolean;                           // not built -> <Placeholder>
}
```

Resolver: `element ?? <Placeholder title={name ?? path} />`. Nav is generated by filtering the same registry on `name && passesPermission && passesCondition`. **Build a page = flip `placeholder` to a real `element`.** Nothing else to touch.

Seed all routes from `V1_UI_Map.md §3` as `placeholder: true` on day one.

## 4. Phase 1 deliverables (definition of done)

1. **Scaffold** builds clean: `vite build` with **zero babel**, Tailwind v4, strict TS passing.
2. **Bootstrap** (`bootstrap.ts`) reads all `window.*` globals + CSRF meta into the Zustand session/theme/flags stores. (Fallback path: a future `/api/client/me` endpoint — note as Phase 2.)
3. **Landing page** — public marketing/entry page; CTA → `/auth/login`; respects theme tokens.
4. **Login page** — email + password + **Turnstile** (conditional on `SiteConfiguration.captcha`), error states, 2FA checkpoint stub route. Hits existing `api/routes/auth/login`.
5. **App shell + navigation** — `TopNav` + the three sidebars (Account / Server / Admin) generated from the registry, with category grouping, active-state, mobile drawer, permission/flag filtering.
6. **Shared `<Placeholder>`** — one component rendered for every not-yet-built route: shows page title, "coming soon" state, breadcrumb, and back nav. Used by ~all routes initially.
7. **Routing** — all four routers wired; unknown path → NotFound; auth guard redirects unauthenticated users to `/auth/login`.
8. **Security baseline** (carry into every phase):
   - CSRF token from `<meta>` on every mutating request (axios interceptor).
   - No `dangerouslySetInnerHTML`; if unavoidable, **DOMPurify**.
   - Strict CSP-friendly (no inline event handlers); keep Turnstile + Stripe origins allow-listed.
   - `httpOnly`/secure cookies stay server-side; never store tokens in `localStorage`.
   - Zod-validate all form input before submit.

**Acceptance test:** log in, land in the dashboard, click through every sidebar entry in all three areas — each unbuilt route shows the shared placeholder, no console errors, build has no babel in the dependency graph.

## 5. Suggested Phase order after this

- **Phase 2** — Account area (overview, credentials, security) + real session via `/me`; the axios/Fractal transformer layer; TanStack Query patterns locked in.
- **Phase 3** — Server core: console (websocket + xterm), files, settings, startup.
- **Phase 4** — Server data: databases, backups, schedules, network, users.
- **Phase 5** — Admin management: nodes, servers, users, roles, nests/eggs.
- **Phase 6** — Modules behind flags: billing, tickets, AI, extensions, theme, webhooks.

Each phase = flip placeholders to real pages in the registry; the shell never changes.
