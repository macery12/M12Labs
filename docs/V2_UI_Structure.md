# V2 UI — Structure at a Glance

The new UI is a **self-contained app in [`ui/`](../ui/)** with its own `package.json`, Vite config and `node_modules` — fully isolated from the V1 build so the two never collide.

## Stack
- **React 19** + **Vite 8** with **`@vitejs/plugin-react-swc`** (Rust/SWC — **no babel**, sub-second builds). All deps tracked at latest.
- **Tailwind v4** (`@tailwindcss/vite`) for styling; design tokens are CSS variables in [`ui/src/styles/tailwind.css`](../ui/src/styles/tailwind.css), with the full semantic theme injected at runtime from `window.ThemeConfiguration` (see **Theming** below).
- **Radix UI** primitives + **CVA** for accessible, variant-driven components ([`ui/src/components/ui`](../ui/src/components/ui)).
- **react-router-dom v7** (routing), **TanStack Query v5** (server data), **Zustand** (global UI state), **react-hook-form + zod** (forms).

## How it's driven
- **Route registry** ([`ui/src/routes/*.routes.ts`](../ui/src/routes/)) is the single source of truth: each entry carries `path / name / icon / category / permission / condition`. It drives **both** routing **and** the sidebars. No `element` → the shared [`Placeholder`](../ui/src/pages/_shared/Placeholder.tsx) renders.
- **Bootstrap** ([`ui/src/app/bootstrap.ts`](../ui/src/app/bootstrap.ts)) reads the `window.*` globals (user, flags, theme, flashes) into Zustand stores — same Blade→JS handoff as V1.
- **Data** comes from the existing Laravel API via [`ui/src/lib/http.ts`](../ui/src/lib/http.ts) (axios + CSRF), e.g. the dashboard fetches `GET /api/client`.

## Server cockpit
- The server page (`/v2/server/:id`) is a **fixed, ops-grade dashboard** (not user-editable): a live metric strip, a **console hero** with a right rail (power / live usage / info), and a network + activity row. Panels live in [`ui/src/pages/server/panels`](../ui/src/pages/server/panels) and share one chrome ([`Panel.tsx`](../ui/src/pages/server/panels/Panel.tsx)) — hairline borders, tight radius, uppercase micro-labels, mono numerals.
- Live data is a **websocket** ([`ui/src/lib/Websocket.ts`](../ui/src/lib/Websocket.ts), Sockette) to Wings: `GET …/websocket` → auth → `stats` / `console output` / `status` streams, surfaced through the [`serverSocket`](../ui/src/state/serverSocket.ts) store. The console is xterm.js; power uses the socket's `set state`.

## Admin infrastructure
- **Nodes and servers are one menu entry**, not two. A single **Infrastructure** section (`/v2/admin/infrastructure`) owns the merged overview and both detail cockpits; the legacy `/v2/admin/nodes` and `/v2/admin/servers` paths (overview **and** `:id`) redirect into it (hidden `nodes/*` / `servers/*` registry entries render [`NodesRedirect`/`ServersRedirect`](../ui/src/pages/admin/infrastructure/InfraRedirect.tsx)). The single nav entry is gated by `['nodes.read','servers.read']` (any-of).
- **Overview** ([`InfrastructureOverviewPage`](../ui/src/pages/admin/infrastructure/InfrastructureOverviewPage.tsx)) is a persisted 3-way **view toggle** over one unified fleet summary strip (nodes / supercharged / servers / memory / disk) and a "New" split button (New server / New node):
  - **Map** (default) — the pannable **network map** ([`NetworkMap`](../ui/src/pages/admin/servers/NetworkMap.tsx)): the panel (site name) is the centre, branching to **node** hubs and then **server** dots. Drag to pan, scroll to zoom (cursor-centred), click a node to focus, double-click a node/server to open it. Server dots are status-coloured with a hover detail card; includes a floating legend / controls / live stats panel, a highlight search, zoom + fit controls, and a minimap. Pure SVG with a CSS-transform pan/zoom group — no graph/physics dependency. (The map carries its own stats panel, so the summary strip is hidden in this view.)
  - **Servers** — a dense table (status / server / node / owner / limits) ([`ServersTable`](../ui/src/pages/admin/servers/ServersTable.tsx)).
  - **Nodes** — the ops-grade **capacity board**: a grid of [`NodeCard`](../ui/src/pages/admin/nodes/NodeCard.tsx)s, each with memory/disk allocation bars (overallocation-ceiling marker), server count, and status badges. There is no geo data on nodes (the `location` FK was dropped), so the board is capacity-driven.
- **Node detail** (`/v2/admin/infrastructure/nodes/:id`, [`NodeDetailPage`](../ui/src/pages/admin/nodes/NodeDetailPage.tsx)) is a tabbed cockpit — **Overview** (live host CPU/mem/disk/swap, auto-polled every ~2.5s, plus allocated capacity and daemon system info), **Allocations**, **Servers**, **Configuration** (read-only daemon config), and a **Wings-RS** tab. **Wings-RS (Supercharged) is gated, not separate** — every node shows the tab; standard nodes get a detect button + "features locked" state, and `wings_type === 'wings-rs'` nodes unlock live system stats (auto-poll), version/build overview, and a log viewer. The gate mirrors the backend's `Node::isSupercharged()`.
- **Server detail** (`/v2/admin/infrastructure/servers/:id`, [`ServerDetailPage`](../ui/src/pages/admin/infrastructure/server/ServerDetailPage.tsx)) is the admin server cockpit — a **single-page editor** ([`ServerEditor`](../ui/src/pages/admin/infrastructure/server/ServerEditor.tsx)) with a sticky in-page **section nav** (scrollspy via `IntersectionObserver`) and a sticky **save bar** that tracks unsaved changes. Loaded from one `GET /servers/{id}?include=egg,node,user,variables,allocations` call (`ServerContext`); the header carries the lifecycle actions (suspend / unsuspend / reinstall / force-delete). Sections:
  - **Information** — name, external identifier, owner, description.
  - **Startup** — nest + egg selectors (changing the egg adopts that egg's defaults), startup command (egg default shown as placeholder), **docker image** as a dropdown of the egg's tagged images (label = the egg's display name, e.g. "Java 21"; a previously-set custom image is preserved as an extra option), skip-install-script toggle, and every **environment variable** (current `server_value`, default as placeholder, description).
  - **Resources** — CPU limit, **CPU pinning** (threads), memory, swap, disk, block-IO, OOM-killer toggle.
  - **Networking** ([`NetworkingSection`](../ui/src/pages/admin/infrastructure/server/NetworkingSection.tsx)) — the assigned allocations with a **primary** marker; set-primary, remove, and add-from-free are **staged** locally and committed on save (`allocation_id` / `add_allocations` / `remove_allocations`).
  - **Feature limits** — allocations, backups, databases, subusers, and **subdomains** (the custom-subdomain limit).
  - One **Save** commits both `PATCH /servers/{id}/startup` (startup / image / egg / env / skip-scripts) and `PATCH /servers/{id}` (details / limits / feature-limits / allocations); a **Discard** resets to the loaded state. The servers-table "Edit" action navigates here. Without `servers.update` the whole page renders read-only.
- **Mutations / management** — the section is fully writable, all gated by `can(held, …)` (forward-compat; admins hold `*` today):
  - **Create node** ([`CreateNodeModal`](../ui/src/pages/admin/infrastructure/CreateNodeModal.tsx), `POST /nodes`) — a grouped modal (Identity / Capacity / Ports / Advanced); the same component runs in **edit** mode seeded from the node detail (`PATCH /nodes/{id}`), launched from the node header. **Delete node** is a confirm dialog from the header (`DELETE /nodes/{id}`, then navigate back to the overview).
  - **Create server** ([`CreateServerModal`](../ui/src/pages/admin/infrastructure/CreateServerModal.tsx)) has two modes: **From preset** (pick preset + deployable node → `POST /servers/preset`; the service auto-assigns owner, the first free allocation, and egg env defaults) and **Manual** (owner → node → free allocation → nest→egg, then the egg seeds docker image / startup / a variables field-set, plus full resource + feature limits → `POST /servers`). Preset CRUD lives **inside** the create flow via an inline [`PresetManager`](../ui/src/pages/admin/infrastructure/PresetManager.tsx) ("Manage presets") over `/servers/presets`.
  - **Server row actions** ([`ServersTable`](../ui/src/pages/admin/servers/ServersTable.tsx) menu) — Suspend / Unsuspend (state-aware) / Reinstall / **Edit** (navigates to the server detail page) / **Delete** (confirm with a force switch → `POST /servers/{id}/delete`). Full server editing — details, build, **and startup/variables** — lives on the server detail page (see above).
  - **Allocations** ([`NodeAllocationsTab`](../ui/src/pages/admin/nodes/panels/NodeAllocationsTab.tsx)) — an add form (ip / alias / port range → `POST /nodes/{id}/allocations`) and per-row delete on **unassigned** allocations (`DELETE …/allocations/{id}`).
  - Forms are built on three new shared primitives — [`Modal`](../ui/src/components/ui/Modal.tsx) (Radix Dialog), [`Select`](../ui/src/components/ui/Select.tsx) (Radix Select) and [`ConfirmDialog`](../ui/src/components/ui/ConfirmDialog.tsx) — with `react-hook-form` validation, optimistic `useFlashes` toasts, and TanStack query invalidation. **No backend changes** — every endpoint already existed.
- All data comes from the session-authed **application API** (`/api/application/nodes/*`, [`getAdminServers`](../ui/src/api/adminServers.ts) → `GET /api/application/servers?include=node,user`, paged to 100) via the same [`http`](../ui/src/lib/http.ts) client; server view-models are normalised once in [`serverState.ts`](../ui/src/pages/admin/servers/serverState.ts). Nested-routed by [`InfrastructureSection`](../ui/src/pages/admin/infrastructure/InfrastructureSection.tsx); cockpits share the ops-grade [`Panel`](../ui/src/components/ui/Panel.tsx) chrome with the server cockpit. **No backend changes** — UI consolidation only.

## Admin billing
- **Navigation lives inside the section, not the main sidebar.** The main admin sidebar keeps a single flat **Billing** entry; [`BillingSection`](../ui/src/pages/admin/billing/BillingSection.tsx) renders its own in-page secondary nav ([`BillingNav`](../ui/src/pages/admin/billing/BillingNav.tsx)) — a grouped rail (Dashboard / Catalog / Transactions / System) on `lg+`, collapsing to a horizontal scroll strip on small screens, each item permission-gated. The focused **product editor** routes render full-width (no rail). Internal routing is a nested `<Routes>`.
- **Overview** (`/v2/admin/billing`, [`BillingOverviewPage`](../ui/src/pages/admin/billing/BillingOverviewPage.tsx)) is the analytics dashboard from one `GET /billing/analytics` call ([`api/billing.ts`](../ui/src/api/billing.ts)): a KPI row (7-day revenue forecast + 30-day, renewals due in 14 days, active products/categories, suspended-server count), an **Upcoming renewals** breakdown (overdue / 7d / 8–14d), a **Recent events** list, and a **Suspended servers** list.
- **Products** (`/v2/admin/billing/products`, [`ProductsPage`](../ui/src/pages/admin/billing/products/ProductsPage.tsx)) is a clean **category list** (a V2 redo of V1's `CategoryContainer` flow). Each row shows the category name, visibility, and its **allowed eggs as named chips** (primary highlighted); clicking the row opens the category detail page. Egg names resolve from one [`getNestEggs`](../ui/src/api/nests.ts) query per distinct nest. **New category** routes to the detail page in create mode; per-row edit/delete (`ConfirmDialog`).
- **Category detail** (`/v2/admin/billing/products/categories/new` and `…/categories/:id`, [`CategoryDetailPage`](../ui/src/pages/admin/billing/products/CategoryDetailPage.tsx)) is a **single sectioned page** (shared editor chrome in [`editorChrome.tsx`](../ui/src/pages/admin/billing/editorChrome.tsx) — section cards + sticky save bar): **Settings** (name, icon, visibility, egg/plan-change permissions), **Egg selection** (nest selector + an allowed-eggs checklist where each egg can be starred **primary**), and an inline **Products** section listing the category's plans with New / edit / delete. Creating a category lands you on its detail page so you can pick eggs and add products. The egg/nest pickers reuse [`nests.ts`](../ui/src/api/nests.ts).
  - **Products are loaded per category** via `GET /billing/categories/{id}/products`. The category→products `include` is **not** used: the backend `Category::products()` `hasMany` omits the `uuid` local key, so it resolves against `categories.id` and returns nothing. Loading per-category sidesteps that without a backend change.
- **Product editor** (`/v2/admin/billing/products/new?category=:id` and `…/products/:id?category=:id`, [`ProductEditorPage`](../ui/src/pages/admin/billing/products/ProductEditorPage.tsx)) is a **dedicated full page** (same shared section-card + save-bar chrome): **Details**, **Limits** (cpu, memory, disk, backup, database, allocation, subdomain), and a **Billing cycles** editor (add/remove/toggle `{days, is_enabled}` rows). Save runs `createProduct`/`updateProduct` then `syncBillingCycles`; create then routes to the new product's editor. The back-link returns to the owning category's detail page.
  - **Payload quirk (intentional):** the product controller *reads* a nested `limits` object while `StoreBillingProductRequest` *validates* flat `*_limit` keys — so [`billingProducts.ts`](../ui/src/api/billingProducts.ts) sends **both** (verified end-to-end). The configured billing currency isn't exposed to the V2 frontend yet, so [`formatCurrency`](../ui/src/lib/format.ts) defaults to USD.
- **The remaining tabs** (Orders, Invoices, Coupons, Exceptions, Settings, Invoice Settings) are **navigable stubs** this round — routed to a themed [`ComingSoon`](../ui/src/pages/admin/billing/ComingSoon.tsx) placeholder, built out in later passes. All data flows through the session-authed application API (`/api/application/billing/*`); **no backend changes**.

## Admin extensions
- **Overview** (`/v2/admin/extensions`) is an **extension gallery** ([`ExtensionsOverviewPage`](../ui/src/pages/admin/extensions/ExtensionsOverviewPage.tsx)): a summary strip (installed / enabled / updates / repositories), a segmented status filter (All / Installed / Available / Updates) + search, and a responsive grid of [`ExtensionCard`](../ui/src/pages/admin/extensions/ExtensionCard.tsx)s. Each card is status-toned (core / enabled / installed / available / update) with a quick **enable toggle** and **Install**/**Manage** action; manifest icons resolve to Lucide glyphs via [`extMeta`](../ui/src/pages/admin/extensions/extMeta.ts).
- Clicking a card opens the **[`ExtensionManageDrawer`](../ui/src/pages/admin/extensions/ExtensionManageDrawer.tsx)** (right slide-over): enable, a **manifest-driven settings-schema form** (text / textarea / number / boolean / select / password), **server-access** controls (allowed nests/eggs checklists from `/extensions/nests-eggs`), plus **Install / Update / Uninstall** with a danger zone.
- A **[`RepositoriesPanel`](../ui/src/pages/admin/extensions/RepositoriesPanel.tsx)** lists manifest sources with a health dot (ok / disabled / error), package counts, enable/disable, and add/edit/delete (the official repo can be disabled but not deleted). Long-running install/uninstall/update operations are surfaced by the **[`OperationProgressBanner`](../ui/src/pages/admin/extensions/OperationProgress.tsx)**, fed by polling `/extensions/progress`; all action buttons lock while an operation runs.
- All data comes from the existing application API (`/api/application/extensions/*`) via [`api/extensions.ts`](../ui/src/api/extensions.ts) — **no backend changes**. Manifest-sourced copy (name, description, author, source label, security warning, settings-field labels) is rendered verbatim and intentionally **not** routed through the i18n catalog; only panel-coded chrome is. Nested-routed by [`ExtensionsSection`](../ui/src/pages/admin/extensions/ExtensionsSection.tsx) under the registry's `extensions/*` entry.

## Theming
- The theme is a **semantic, layered model** resolved once in [`ui/src/lib/theme.ts`](../ui/src/lib/theme.ts) and written onto CSS variables by `applyThemeVars()`:
  - **Brand** — a single `primary`; hover/soft tints are **derived** via `color-mix` so changing the brand "just works" (the old bug where `--brand-hover` stayed fixed is gone).
  - **Base** — `canvas / surface / surface_2 / border / ink / ink_muted` as a coherent ramp (subtle border + faint-text tints derive automatically). Choose a **named base palette** (`BASE_PALETTES`) or override tokens individually.
  - **Status** — `accent / warning / danger`.
  - **Feel** — corner `radius`, the background `.bg-grid` (on/off + line strength + cell size) and the ambient `.bg-aurora` glow (on/off + intensity), all driven by CSS vars + `data-grid`/`data-aurora` attrs.
- **Default is M12Labs Blue** (`#0047fc`) — both the CSS fallback and the backend module default ([`config/modules/theme.php`](../config/modules/theme.php)).
- **Admin page** `/v2/admin/theme` ([`ThemeSection`](../ui/src/pages/admin/theme/ThemeSection.tsx)) edits a **draft** with a sandboxed **live preview** ([`ThemePreview`](../ui/src/pages/admin/theme/ThemePreview.tsx) — same `applyThemeVars` scoped to a container) and only persists on **Save**. Built-in + custom **presets** (color schemes) apply into the draft; "Save current" snapshots the committed colors as a new preset.
- **Backend**: the generic `theme` key/value table stores overrides (`theme::colors:*`, `theme::feel:*`); [`ThemeServiceProvider`](../app/Providers/ThemeServiceProvider.php) loads them into config and [`ThemeComposer`](../app/Http/ViewComposers/ThemeComposer.php) emits `{ colors, feel }`. The V2 client uses [`api/theme.ts`](../ui/src/api/theme.ts) over `/api/application/theme` (`PUT /colors`, `POST /reset`, preset CRUD + apply). Presets live in `theme_presets`; the expanded built-ins are seeded by [the refresh migration](../database/migrations/2026_06_28_000001_refresh_theme_presets_expanded_palette.php).

## Localization (i18n)
- All user-facing copy is externalized through **`i18next` + `react-i18next`** (init in [`ui/src/i18n/index.ts`](../ui/src/i18n/index.ts)). **English is the source of truth** and the only shipped locale today; the model is built so Crowdin + more languages drop in later with no UI changes.
- Catalogs are **namespaced per feature** — `ui/src/i18n/locales/en/<namespace>.json` (e.g. `common.json`, `dashboard.json`), registered in [`resources.ts`](../ui/src/i18n/resources.ts). `common` is the default namespace (shared `actions.*` / `states.*` / `nav.*`). Components call `useTranslation('<ns>')` → `t('key')`, interpolating `{{name}}` (React-escaped, so `escapeValue:false`).
- **Keys are type-checked.** [`i18n.d.ts`](../ui/src/i18n/i18n.d.ts) augments i18next's `CustomTypeOptions` from the English catalog, so `t('unknown.key')` is a **compile error** — `pnpm build` (`tsc -b`) fails if a page references a string that isn't in the catalog. This is what keeps the catalog and the UI from drifting.
- Initial locale resolves from (localStorage choice → `window.PterodactylUser.language` → `window.SiteConfiguration.locale` → `en`), keeping `<html lang>` in sync; `setLocale()` is exposed for a future language picker. All current V2 pages/components are externalized (namespaces: `common`, `nav`, `auth`, `landing`, `dashboard`, `server`, `admin`, `extensions`).
- **Crowdin**: [`crowdin.yml`](../crowdin.yml) at the repo root maps the English catalog to per-locale sibling folders. Translation flow is `crowdin upload sources` → translate → `crowdin download`; a downloaded locale is registered in [`resources.ts`](../ui/src/i18n/resources.ts) + added to `SUPPORTED_LOCALES`.

## Serving
Laravel serves it at **`/v2`** (web-only route → [`templates/v2/core.blade.php`](../resources/views/templates/v2/core.blade.php)); the SPA owns all client-side routing under that prefix.
